A literate approach to configuration gives it a more personal taste. Somewhat stolen from angrybacon Note that this is not ideal for maintainability, for this see purcell’s emacs.d for its small modules. No, this is for me to apply my arbitrary ontology, encoding my understanding of Emacs! Or at least what I want from it.

In the proper LISP fashion it’ll be done in a ouroboros. Org will contain Emacs’s configuration, which configures Org. Emacs(.d) will also logically contain the body of my Org files, including <~/org/blog> which will define my Hugo generated, (github hosted) website, which this literate configuration ought fit on verbatim…

I went off Hugo a while to check out the org-static-blog which is nice, but for what I need just the better look of a Hugo generated site is sufficient.
